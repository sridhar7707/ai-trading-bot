# TradeGenius AI — Design Principles

Last updated: 2026-06-27

These principles were established during the architecture review (2026-06-14) and govern
all future development decisions.

## 1. Safety First, Features Second

Every trading decision is gated through `RiskManager` before reaching `AlpacaClient`.
No code path may bypass this gate. The risk constants in `config.py`
(`STOP_LOSS_PCT`, `DAILY_LOSS_LIMIT_PCT`, `MAX_POSITION_PCT`) require an explicit
architecture review before changing.

## 2. Explainability Is Not Optional

Every BUY and SELL signal must carry human-readable reasons. SHAP drivers (stored in the
`trades` table `feature_drivers` column) power the Telegram alerts and the Dashboard Why
panel. A signal without an explanation is treated as an implementation gap.

## 3. Layered Architecture, One-Way Data Flow

Data flows strictly downward through 6 layers:

```
Layer 1: Data Ingestion   (features.py, macro.py, sentiment.py)
Layer 2: Regime           (regime_classifier.py)
Layer 3: Signal           (xgb_predictor.py, lstm_predictor.py, ensemble.py)
Layer 4: Risk             (risk_manager.py)
Layer 5: Execution        (alpaca_client.py)
Layer 6: Monitoring       (dashboard, telegram, sync_db)
```

Upper layers must never import lower layers. `alpaca_client` is only called by Layer 4/5
code, never directly from strategy or data layers.

## 4. Single Source of Truth

- Trading parameters: `config.py` only — no hardcoded constants elsewhere
- Recommendation logic: `bot/core/recommendation_engine.py` — all dashboard panels call
  the same 5 helper functions
- Design tokens: `dashboard/design_system.py` — all colors, fonts, and spacing
- Requirements state: `tests/req_snapshots/req_state.json` — REQUIREMENTS.md is derived

## 5. File Size Limit: 500 Lines

No file may exceed 500 lines. When a file approaches this limit, extract coherent
functional groups into new modules before continuing. This keeps review cost low and
prevents "god modules".

## 6. No Secrets in Code

All credentials are environment variables loaded via `config.py` / `dotenv`. The
`.env` file is in `.gitignore`. CI secrets live in GitHub Actions repository secrets.

## 7. Fail Safe, Log Everything

All render functions are wrapped with `@safe_render` (returns an error card, never
crashes the dashboard). All bot-cycle exceptions are logged via `log_exception()`.
Silent failures (`except: pass`) are a `[BLOCK]`-level architecture violation.

## 8. Paper Trading First

The bot runs against Alpaca paper trading (`https://paper-api.alpaca.markets`).
The base URL is environment-configurable but defaults to paper. A live-money switch
requires an explicit `.env` override and cannot happen by accident.

## 9. Tests for Critical Paths

Every function on the order execution path (risk check → size → execute) must have
unit tests. Dashboard render functions are tested by `ui_tester.py`. New bot modules
require a `tests/test_<module>.py` file.

## 10. Docs Updated Before Merge

`docs/REQUIREMENTS.md` is regenerated by `tests/requirements_tracker.py` after every
change. `docs/UI_CHANGELOG.md` is updated by `tests/ui_changelog.py`. Both are
checked-in artifacts, not gitignored.
